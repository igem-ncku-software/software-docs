# 網頁工具部署上線的紀錄

這篇文章記錄了我們 iGEM 軟體工具從本機開發到正式上線的完整過程：前端用 HTML/CSS/JS，
後端用 FastAPI，前端部署到 GitHub Pages，後端部署到 Render，並解決了部署過程中遇到的
幾個實際問題（例如 GitHub Pages 資料夾限制、Jekyll 蓋掉自訂頁面等）。

<!-- more -->

## 之後要更新網站時的操作流程（先記住這個）

之後不管是改前端樣式、改後端分析邏輯，流程都一樣，**兩邊都會自動重新部署**，不需要手動做任何額外的事：

```bash
# 1. 修改完程式碼後，在專案根目錄執行
git add .
git commit -m "說明這次改了什麼"
git push
```

推送之後：

| 修改的資料夾 | 誰會自動重新部署 | 大概要等多久 |
|---|---|---|
| `frontend/` | GitHub Actions 會自動重新打包部署到 GitHub Pages | 1-2 分鐘 |
| `backend/` | Render 會自動偵測並重新部署 | 2-5 分鐘 |

**檢查更新有沒有成功：**

- 前端：GitHub repo 的 **Actions** 分頁，看「Deploy frontend to GitHub Pages」這個 workflow 有沒有跑出綠色勾勾
- 後端：Render 專案頁面的 **Logs**，看有沒有出現 `Your service is live 🎉`

**測試前記得強制重新整理瀏覽器**（`Ctrl+Shift+R` / `Cmd+Shift+R`），避免看到舊版本的快取畫面。

**如果改的是 CORS 設定**（`backend/main.py` 裡的 `allow_origins`），要注意網址不能打錯：
只到網域那一層（例如 `https://igem-ncku-software.github.io`），不要加後面的路徑，結尾也不要加斜線。

## 專案架構

```
igem-webapp/
├── .github/
│   └── workflows/
│       └── deploy-pages.yml     # 自動部署前端用的 GitHub Actions 設定
├── frontend/                    # 純靜態網頁，部署到 GitHub Pages
│   ├── index.html
│   ├── style.css
│   └── script.js
├── backend/                     # FastAPI 後端，部署到 Render
│   ├── main.py
│   └── requirements.txt
├── .gitignore
└── README.md
```

整體概念：

```
使用者瀏覽器 (HTML+CSS+JS)  --fetch()-->  FastAPI 後端 (Python)
   ↓ 純靜態檔案                              ↓ 處理文字/圖片、回傳 JSON
GitHub Pages（免費）                    Render（免費方案可跑 Python）
```

關鍵觀念：**GitHub Pages 只能放靜態檔案，不能跑 Python**，所以前端和後端從一開始就要規劃成兩個獨立部署的服務，靠前端的 JavaScript 用 `fetch()` 呼叫後端 API 溝通。

---

## 1. 本機開發與測試

先在本機把兩邊都跑起來確認邏輯正確，再考慮上線。

### 後端

```bash
cd backend
python -m venv venv
source venv/bin/activate      # Windows 用 venv\Scripts\activate
pip install -r requirements.txt
python main.py
```

啟動後打開 `http://127.0.0.1:8000/health`，看到 `{"status":"ok"}` 代表成功。
FastAPI 會自動產生一個測試介面在 `http://127.0.0.1:8000/docs`，可以直接在網頁上測試
`/api/analyze` 這個 API，不用自己寫測試程式。

### 前端

直接用瀏覽器打開 `frontend/index.html`，確認 `script.js` 裡的 `BACKEND_URL` 指向
`http://127.0.0.1:8000`。輸入文字、選圖片、按送出，確認能拿到後端回傳的 JSON 結果。

### 為什麼虛擬環境建在 `backend/` 而不是根目錄

因為只有 `backend/` 需要 Python 套件（`requirements.txt` 也只在這裡）；`frontend/`
是純 HTML/CSS/JS，瀏覽器直接讀取，完全不需要虛擬環境。這也對應到後面部署時，
Render 的 **Root Directory** 設定會指定成 `backend`，只在這個資料夾裡安裝套件、啟動服務。

## 2. 推上 GitHub

在專案根目錄（同時看得到 `frontend/`、`backend/`、`.gitignore` 的那一層）執行：

```bash
git init
git add .
git commit -m "Initial commit: iGEM analyzer"
git branch -M main
git remote add origin https://github.com/igem-ncku-software/igem-software.git
git push -u origin main
```

推上去之後回到 GitHub 網頁確認：檔案都在，且**看不到 `venv/` 資料夾**——這代表
`.gitignore` 有正常排除虛擬環境、快取檔案等不該進版控的東西。

`.gitignore` 內容大致涵蓋：

```
# Python
__pycache__/
*.py[cod]

# 虛擬環境
venv/
.venv/

# 環境變數 / 機密資訊
.env
*.key

# IDE / 系統檔案
.vscode/
.idea/
.DS_Store
```

## 3. 部署後端到 Render

1. [render.com](https://render.com) 用 GitHub 帳號登入並授權
2. **New +** → **Web Service** → 選擇連接的 repo
3. 關鍵設定：

| 欄位 | 填入 |
|---|---|
| Root Directory | `backend` |
| Build Command | `pip install -r requirements.txt` |
| Start Command | `uvicorn main:app --host 0.0.0.0 --port $PORT` |
| Instance Type | Free |

4. Environment Variables 這一步**可以先跳過**——目前程式碼裡沒有任何需要保密的
   API 金鑰或資料庫連線字串，全部設定都直接寫在程式碼裡就能跑。以後如果串接
   外部付費 API，才需要回來這裡加變數。

5. 建立後等待 build log 跑完，出現 `Your service is live 🎉` 代表成功，會拿到一個網址：
   ```
   https://igem-ncku-software.onrender.com
   ```

打開 `/health` 和 `/docs` 兩個路徑確認後端正常運作。

!!! note "免費方案的小提醒"
    Render 免費方案的服務閒置一段時間會「睡著」，下次有請求進來時，
    第一次回應可能要等 30-50 秒喚醒，屬於正常現象，之後就會恢復正常速度。

## 4. 部署前端到 GitHub Pages

### 一開始踩到的坑：資料夾限制

用「Deploy from a branch」這個方式時，GitHub 網頁介面**只能選 `/(root)` 或 `/docs`**，
不能自訂成 `/frontend`。解決方式選擇改用 **GitHub Actions** 部署，這樣可以保留原本
`frontend/` 的資料夾名稱，不用改名。

### 建立 workflow 檔案

`.github/workflows/deploy-pages.yml`：

```yaml
name: Deploy frontend to GitHub Pages

on:
  push:
    branches: ["main"]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Setup GitHub Pages
        uses: actions/configure-pages@v5

      - name: Upload frontend folder as artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: "./frontend"

      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

推上去後，到 **Settings → Pages**，把 **Source** 從 `Deploy from a branch` 改成
**`GitHub Actions`**。

### 第二個坑：Jekyll 蓋掉自訂頁面

切換 Source 的過程中，舊的「Deploy from a branch」流程（`pages build and deployment`）
跟新的 Actions workflow 幾乎同時被觸發，導致 GitHub 內建的 Jekyll 引擎把 `README.md`
渲染成首頁，蓋掉了原本 `frontend/index.html` 的內容。

**解法：** 用一個空白 commit 強制重新觸發 Actions 部署一次：

```bash
git commit --allow-empty -m "Trigger Pages redeploy"
git push
```

跑完之後強制重新整理瀏覽器（`Ctrl+Shift+R` / `Cmd+Shift+R`）清除快取，頁面就恢復正常了。

最終網站網址：
```
https://igem-ncku-software.github.io/igem-software/
```

## 5. 收緊 CORS，完成上線

一開始為了方便測試，後端 CORS 設定是完全開放：

```python
allow_origins=["*"]
```

前後端都測試成功之後，把它收緊成只允許自己的 GitHub Pages 網址呼叫，比較安全：

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["https://igem-ncku-software.github.io"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

推上去後 Render 會自動偵測到 `backend/` 有變動並重新部署，等 Live 之後重新整理前端網站，
確認右下角 **Backend status** 顯示「已連線 ✅」，實際跑一次文字 + 圖片送出測試，
確認能拿到分析結果，就代表整個流程正式完成。

## 小結：踩過的坑整理

| 問題 | 原因 | 解法 |
|---|---|---|
| GitHub Pages 選不到 `/frontend` | 內建的「Deploy from a branch」只支援 `/(root)` 和 `/docs` | 改用 GitHub Actions 部署 |
| 網站打開變成 README 內容 | 舊的 Jekyll 流程跟新的 Actions 部署衝突，Jekyll 跑得比較晚蓋掉結果 | 空白 commit 強制重新觸發 Actions |
| 前端顯示「連不上後端」 | CORS 設定跟前端網址不一致，或後端還在睡眠喚醒中 | 檢查 `allow_origins` 網址是否精確、耐心等待 30-50 秒 |

之後如果要繼續擴充這個工具（例如接真正的序列分析、模型推論），核心的
前後端串接、CI/CD 部署流程都已經打通，只需要專注在 `backend/main.py`
裡的分析邏輯本身即可。
